1. Project Overview
MedExplainer AI is a cloud-native, HIPAA-aligned platform that ingests patient medical reports and generates personalized, multimodal educational videos — combining AI-driven narration, 3D body visualizations, and simplified text — enabling doctors to improve patient comprehension and treatment adherence.

Problem
Doctors lack time to explain complex conditions; patients forget or misunderstand verbal explanations.
Solution
Automatically generate a personalized video explainer from a patient's own medical data using AWS AI services.
Primary Users
Doctors (consultation aid) and Patients (self-education post-visit).
Target Market
Clinics, hospitals, and telemedicine platforms across India and emerging markets.
Compliance
HIPAA-aligned data handling, end-to-end encryption, zero data retention on inference nodes.


2. System Architecture
2.1 High-Level Data Flow
The system follows a strict unidirectional pipeline to maintain data integrity and patient confidentiality:

End-to-End Pipeline
Step 1  →  Doctor uploads medical report (PDF / HL7 FHIR / structured lab data)
Step 2  →  API Gateway + AWS WAF receive request over TLS 1.3
Step 3  →  Amazon Textract extracts text, tables, and key-value pairs from the report
Step 4  →  Amazon Comprehend Medical identifies clinical entities (conditions, medications, dosages)
Step 5  →  Amazon Bedrock (Claude) personalizes explanation — patient age, language, literacy level
Step 6  →  Amazon Polly synthesizes audio narration in Hindi / English / regional language
Step 7  →  AWS Lambda orchestrates media stitching (body diagram overlays + narration + text)
Step 8  →  Amazon S3 (encrypted) stores the generated video temporarily (TTL: 24 hours)
Step 9  →  Doctor / Patient accesses video via pre-signed S3 URL (time-limited, single-use)
Step 10 →  Amazon CloudFront delivers video with geo-restriction and signed cookies

2.2 Microservices Architecture
The backend is decomposed into independently deployable microservices, each running as a containerized workload on Amazon ECS (Fargate):

ingestion-service
Validates, sanitizes, and routes uploaded medical documents. Enforces file-type whitelist and malware scan via Amazon GuardDuty.
extraction-service
Calls Amazon Textract and Comprehend Medical. Normalizes output into a structured JSON schema.
explanation-service
Invokes Amazon Bedrock with patient profile context. Generates patient-friendly explanation in requested language.
media-service
Stitches narration audio (Polly), body SVG/3D overlays, and text captions into MP4 video via FFmpeg Lambda layer.
delivery-service
Generates pre-signed URLs, manages TTL, logs access events to CloudTrail.
auth-service
Issues JWT tokens via Amazon Cognito. Handles doctor and patient role-based access control (RBAC).

4. Security & Confidentiality Architecture
Given the sensitive nature of medical data, security is a first-class design concern — not an afterthought. Every layer of the stack applies the principle of least privilege, zero trust, and defence-in-depth.

4.1 Data Encryption
In Transit
TLS 1.3 enforced on all endpoints. HTTPS-only policy on CloudFront. API Gateway rejects HTTP connections.
At Rest (S3)
AES-256 server-side encryption via AWS KMS CMK. Bucket versioning + Object Lock (WORM) on medical reports.
At Rest (DB)
Amazon Aurora Transparent Data Encryption (TDE) using KMS-managed keys.
Key Rotation
KMS keys auto-rotated annually. Secrets Manager rotates DB credentials every 30 days.

4.2 Access Control
Amazon Cognito user pools: separate pools for Doctors and Patients with different permission scopes.
Role-Based Access Control (RBAC): doctors can upload + view; patients can view only their own video.
Pre-signed S3 URLs: time-limited (1 hour), IP-bound, single-use — video never directly exposed.
IAM roles follow least-privilege: each Lambda/ECS service has only the permissions it needs.
No cross-service wildcard policies (no s3:* or iam:*); all resources are scoped by ARN.

4.3 Network Security
All compute (ECS, Lambda, RDS) runs inside a private VPC subnet — no public IP addresses.
AWS WAF on API Gateway: blocks SQL injection, XSS, path traversal, and rate-limits each IP to 100 req/min.
AWS Shield Standard: always-on DDoS protection for public-facing endpoints.
Amazon GuardDuty: continuous threat intelligence, anomaly detection on API patterns and S3 access.
VPC Flow Logs + CloudTrail: every network event and API call is logged and immutable.

4.4 Data Minimization & Retention
Patient Data Handling Policy
• Medical reports are stored encrypted in S3 for processing only — deleted after 7 days (configurable lifecycle rule).
• Generated videos are ephemeral — TTL of 24 hours; automatically expired by S3 lifecycle policy.
• Amazon Bedrock inference: NO persistent storage of patient data on model endpoints (on-demand mode).
• Amazon Comprehend Medical: API call only — no training on patient data.
• RDS stores only anonymized metadata (session IDs, hashed report fingerprints) — never raw PII.
• Patients are identified by session token only; no name/DOB stored in application database.

4.5 Audit & Compliance
AWS CloudTrail: every API call (who, what, when, from where) stored in a separate, tamper-proof S3 bucket.
Amazon CloudWatch Alarms: alert on unauthorized access attempts, unusual data volume, and failed auth.
Security Hub: centralized compliance dashboard covering CIS AWS Foundations Benchmark.
HIPAA alignment: architecture follows AWS HIPAA Eligible Services guidelines and BAA requirements.

6. Production Readiness Checklist
6.1 Scalability
ECS Fargate auto-scales based on CPU/memory; Target Tracking Scaling Policy configured.
SQS queue buffers video generation jobs — system handles traffic spikes without data loss.
CloudFront caches static assets; S3 scales infinitely for video storage.
Amazon Aurora Serverless v2 auto-scales DB capacity from 0.5 to 64 ACUs.

6.2 Reliability & High Availability
Multi-AZ deployment: ECS tasks and RDS replicas across 2+ Availability Zones.
Step Functions retry policies: automatic retry with exponential backoff on transient failures.
Dead-letter queues (DLQ) on SQS: failed jobs captured for manual review without data loss.
Health checks on all ECS services; unhealthy containers auto-replaced by Fargate.

6.3 Observability
Structured JSON logging from all services — queryable via CloudWatch Logs Insights.
Distributed tracing via AWS X-Ray — end-to-end latency visibility across the entire pipeline.
Business metrics dashboard: videos generated/day, avg. processing time, error rate, language breakdown.
PagerDuty / SNS alerts on P0 alarms (pipeline failure, auth anomaly, error rate > 1%).

6.4 Cost Management for AWS Credits
AWS Credits Strategy
• Use Bedrock on-demand pricing — no provisioned throughput commitment needed at hackathon scale.
• Lambda + Fargate Spot: use Spot capacity for media-service tasks (80% cost saving on video generation).
• S3 Intelligent-Tiering: auto-move rarely accessed objects to cheaper storage tiers.
• CloudFront Free Tier: first 1 TB/month data transfer free — sufficient for demo scale.
• Set AWS Budgets alert at 80% of credit balance to avoid overspend.
• Aurora Serverless v2 scales to 0 during idle periods — no cost when not processing.

8. API Design (Key Endpoints)

POST /reports/upload
Doctor uploads medical report. Returns job_id for async processing. File validated, scanned, encrypted before storage.
GET /jobs/{job_id}/status
Poll processing status: QUEUED → EXTRACTING → GENERATING → ASSEMBLING → READY → EXPIRED.
GET /videos/{video_id}/url
Returns time-limited (1h), IP-bound pre-signed URL to access generated video. Auth required.
POST /patients/{id}/profile
Set patient language preference, age group, and literacy level for personalized explanation.
GET /audit/logs
Admin-only: retrieve CloudTrail events for a given patient session. Compliance endpoint.
DELETE /reports/{id}
Manually trigger early deletion of a report before the 7-day TTL. Compliance right-to-erasure.

9. Amazon Bedrock — Prompt Engineering Strategy
The quality of patient explanation depends heavily on prompt design. We use a structured system prompt + dynamic user prompt approach:

System Prompt (fixed — loaded from Secrets Manager)
You are a compassionate medical educator. Your task is to explain a patient's medical condition in simple,
clear language that a person with no medical background can understand. Always:
  • Use everyday analogies to explain body parts or processes.
  • Avoid medical jargon; if you must use a term, define it immediately.
  • Structure your explanation in 3 parts: What is happening, Why it matters, What to do next.
  • Keep the tone warm, reassuring, and factual. Never cause unnecessary alarm.
  • Output in the requested language. Target reading level: Grade 6.

Dynamic User Prompt (generated per patient)
Patient Profile: Age 45, Language: Hindi, Literacy Level: Basic
Clinical Findings (from Comprehend Medical):
  - Condition: Type 2 Diabetes Mellitus (HbA1c: 8.2%)
  - Medication: Metformin 500mg twice daily
  - Lab: Fasting blood glucose 210 mg/dL (normal: 70-100)

Generate a personalized explanation following the 3-part structure above.
Include one simple analogy for blood sugar regulation.
Output format: JSON with keys: what_is_happening, why_it_matters, what_to_do, analogy, audio_script
